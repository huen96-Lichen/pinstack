# PinStack v2.5.3 Screenshot Permission Hotfix

## Summary
- Version: `2.5.3`
- Scope: screenshot permission hotfix only
- Goal: fix the blocking case where macOS screen recording permission is already enabled, but PinStack still marks screenshot capture as unavailable and blocks the built-in capture flow.

## Root Cause
The previous permission model used two signals that were not aligned with the real screenshot execution path:

1. `systemPreferences.getMediaAccessStatus('screen')`
2. `desktopCapturer.getSources(...)`

But PinStack's built-in screenshot flow actually executes the macOS `screencapture` command.

This created two failure modes:
- system status could lag behind the actual runtime state;
- `desktopCapturer` could be overly strict, especially when thumbnail generation was empty or degraded, and incorrectly report failure even though `screencapture` was usable.

Result: users could have permission enabled in System Settings and still be blocked from using PinStack's own screenshot feature.

### Why recording worked but screenshot did not
- Recording and screenshot do **not** use the same chain.
- Recording uses the renderer path:
  - `navigator.mediaDevices.getDisplayMedia(...)`
  - `MediaRecorder`
- Screenshot uses the main-process path:
  - `beginRegionScreenshotCapture()`
  - `canProceedWithScreenCapture()`
  - `captureScreenshotAndPin()`
  - actual capture via `desktopCapturer` first, then `screencapture` fallback
- Before this hotfix, screenshot could be blocked by the preflight gate even when the real screenshot command path was still usable.
- That is why the same app instance could record successfully while the built-in screenshot flow still failed.

## Fix Strategy
### 1. Real screenshot-path probe
- Added a lightweight `screencapture` probe.
- The probe writes a tiny temporary image, validates the output, then cleans it up.
- This probe now represents the real screenshot capability.

### 2. Desktop capturer downgraded to diagnostics
- `desktopCapturer` is still checked, but only as supporting diagnostics.
- Thumbnail emptiness no longer counts as a hard failure.
- `sourceCount > 0` is enough to treat the desktop-capturer layer as available.

### 3. Screenshot gate unblocked
- The screenshot execution gate now prioritizes the real screenshot probe.
- If `screencapture` works, PinStack no longer blocks the screenshot flow even if the system permission status appears stale.
- The UI now marks this as a recoverable state instead of a hard denial.

### 4. Diagnostics made explicit
- Permission diagnostics now show:
  - `systemStatus`
  - `screenshotProbe`
  - `desktopProbe`
  - probe errors when available
- Development diagnostics also include a copyable log with:
  - `appPath`
  - `bundleId`
  - raw `getMediaAccessStatus('screen')`
  - accessibility state
  - current capture mode
  - screenshot trigger / execution path / command
  - failure reason and stack
- This makes later investigation possible without guessing.

## Validation
Run:
- `npm run typecheck`
- `npm run build`
- `npm run test`

Manual verification:
1. Launch `/Applications/PinStack.app`
2. Confirm screen recording permission is enabled in macOS System Settings
3. Open the permission panel and click `刷新状态`
4. Confirm screen capture is no longer hard-blocked
5. Trigger built-in screenshot from Capture Hub
6. Confirm the screenshot flow starts normally

## Out of Scope
This hotfix does **not** address:
- tray icon issues
- dock/finder/cmd+tab icon issues
- app signing / notarization
