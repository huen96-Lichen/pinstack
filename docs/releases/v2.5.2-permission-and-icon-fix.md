# PinStack v2.5.2 修复说明（权限可信化 + 系统图标链路修正）

发布日期：2026-04-04  
基线版本：v2.5.0  
目标版本：v2.5.2

## 1. 本次修了哪些问题

### Bug 1：系统里已开启权限，但 PinStack 仍显示未授权
- 将屏幕录制权限判断从单一系统状态读取，升级为“系统状态 + 真实能力探测 + 实例诊断”的联合模型。
- 屏幕录制现在会同时记录：
  - `systemStatus`
  - `probeStatus`
  - `probeError`
  - `recommendedAction`
- 当系统状态滞后、但实际屏幕捕获能力可用时，不再继续一律显示“未授权”。
- 权限卡和诊断区现在能够区分：
  - 确实未授权
  - 已授权但状态未同步
  - 当前运行实例可能不是系统已授权的那个
  - 当前从非稳定安装路径运行

### Bug 2：菜单栏图标显示为白色方块
- 替换了不适合 macOS 菜单栏的旧 tray 资源。
- 现在优先加载真正的 template 资产：
  - `pinstack-menubar-template.png`
  - `pinstack-menubar-template@2x.png`
- 并继续保留 SVG fallback 与加载日志。

### Bug 3：Dock / Finder / Cmd+Tab 图标不稳定
- 补齐运行时 Dock 图标候选路径，优先检查打包产物中的 `Contents/Resources/icon.icns`。
- 打包配置继续固定到正式 `.icns`：
  - `assets/icons/app/pinstack-app-icon.icns`
- 验证链路改成以正式 `.app` 产物为准，而不是以前端资源推断。

## 2. 根因分析

### Bug 1 根因
- 之前只依赖 `systemPreferences.getMediaAccessStatus('screen')`，这会把“系统状态滞后”“未签名实例差异”“从 DMG 或非 /Applications 目录运行”全部压缩成一个笼统的“未授权”。
- 截图执行前也直接信这个单一状态，所以即使系统里看起来已勾选，也可能继续被阻塞。

### Bug 2 根因
- 原菜单栏资源本身不适合 menu bar 小尺寸 template 场景，形体更接近白色小方块而不是状态栏图标。
- 同时旧链路虽然有 fallback，但没有真正解决素材本体的问题。

### Bug 3 根因
- 正式 App Icon 的打包资源链路基本存在，但运行时 Dock 图标候选路径没有优先覆盖正式打包产物里的 `icon.icns`。
- 另外，未签名应用从 DMG 内或非稳定路径运行时，系统图标与权限都更容易表现出“像另一个实例”的问题。

## 4. 验证结果

### 权限链路
**如何验证**
- 从 `/Applications/PinStack.app` 启动正式打包版。
- 打开权限诊断区，查看：
  - App 路径
  - App 包路径
  - bundleId
  - systemStatus / probeStatus
  - installLocationStable
- 关闭屏幕录制权限后尝试截图。
- 再开启权限，返回 PinStack 后测试自动刷新和手动刷新。

**验证结果**
- 状态不再只显示单一“未授权”。
- 已能区分“系统未授权”“状态未同步/需重启”“安装路径不稳定”“实例可能不一致”。
- 当探测显示实际可用时，Capture Hub 不再直接把截图主流程卡死。

### Tray Icon
**如何验证**
- 启动应用后观察 macOS 菜单栏。
- 检查日志中 `[tray.icon]` 是否命中 template PNG 或 fallback。

**验证结果**
- tray 资源已替换为真正的 template 素材。
- 不再沿用旧的方形轮廓资源。
- 仍需在真实深浅模式菜单栏中做肉眼验收。

### Dock / Finder / Cmd+Tab
**如何验证**
- package 后检查：
  - `Contents/Resources/icon.icns`
  - `Info.plist`
- 从 `/Applications/PinStack.app` 启动正式产物，观察 Dock / Finder / Cmd+Tab。

**验证结果**
- 运行时候选路径已优先覆盖正式产物里的 `.icns`。
- 打包配置与运行时链路已对齐。
- 最终显示仍需按正式 `.app` 产物在目标机器上做人工验收。

## 5. 为什么同一台电脑 dev 能用、DMG 后反而不行
- 这不是前端 UI 自身的问题，而是 macOS 权限与实例识别机制的问题。
- 在**无签名前提**下：
  - 开发态运行的 Electron 实例
  - 打包后的 `.app`
  - 从 DMG 内直接运行的 translocated `.app`
  可能会被系统当成不同实例。
- 因此同一台机器上出现“开发版能用、打包版不行”并不奇怪。
- 本轮已经把这件事显式写进诊断与提示，避免继续让用户误以为只是“权限没开”。
- 本轮默认测试要求也固定为：
  - **复制到 `/Applications` 后再启动**，不从 DMG 内直接运行。

## 6. 仍未解决的问题
- 当前按无签名前提处理，无法从根本上消除 macOS 对不同路径/不同实例的 TCC 差异。
- 如果后续要把权限体验做到更稳定，下一步应考虑：
  - 代码签名
  - 稳定安装路径
  - 更正式的发布流程
